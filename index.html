Anatomía de una llamada al modelo

Cuando tu agente le manda una consulta al LLM, el modelo no recibe solo lo que escribió el usuario. Recibe un paquete que vos armaste: instrucciones, historial, resultados de herramientas, fragmentos recuperados. Todo eso se cobra como input, en cada turno.

Las seis fugas

De dónde sale el gasto invisible

Cuando Alberto dice "esta consulta no era grande pero gastó un montón", casi seguro la fuga está en uno de estos seis lugares. Cada uno se diagnostica distinto y se arregla distinto.

01 · fuga

System prompt inflado

Instrucciones largas, definiciones de tools con schemas verbosos, ejemplos few-shot, reglas de negocio. Se paga en cada llamada, aunque el usuario haya escrito tres palabras.

Detección: medí el campo usage.input_tokens en una llamada con input mínimo del usuario. Si ya da 5k+, es esto.
impacto típico
02 · fuga

Tool outputs verbosos

Una query SQL que vuelve con 500 filas, una API que retorna JSON con 80 campos por registro. El modelo recibe todo aunque solo necesite tres datos. Es lo más fácil de detectar y de arreglar.

Detección: logueá el tamaño en bytes de cada tool response antes de mandarla al modelo. Cualquier cosa arriba de 5KB merece revisión.
impacto típico
03 · fuga

Historial acumulado

Si la conversación tiene memoria y van 15 turnos, el turno 16 reenvía los 15 anteriores enteros. El costo total de la sesión crece cuadráticamente aunque cada turno parezca chico.

Detección: graficá input_tokens contra número de turno en una sesión real. Si sube en pendiente, es esto.
impacto típico
04 · fuga

RAG sobredimensionado

Top-20 chunks de 500 tokens = 10.000 tokens de contexto por consulta. Muchas implementaciones traen demasiado "por las dudas" cuando top-5 con buen umbral de similitud rinde igual o mejor.

Detección: mirá tu config de retriever. k arriba de 10 sin tuning explícito es sospechoso.
impacto típico
05 · fuga

Reintentos y loops silenciosos

Si el agente falla un tool call (JSON mal formado, schema invalido, timeout), el framework reintenta. Pagás dos, tres, cinco veces la misma consulta. LangGraph y Agno son los peores ofensores porque sus loops de "self-correction" pueden iterar 8-10 veces antes de rendirse, y todo eso se cobra.

Detección: en logs, buscá patrones tipo tool_use → tool_result (error) → tool_use → tool_result (error) repetidos. Contá llamadas a /v1/messages por interacción del usuario — debería ser ~1, no 3-5. En LangGraph activá recursion_limit bajo (5) y revisá los RecursionError.
impacto típico
06 · fuga

Thinking / extended reasoning

Si activaste modelos con thinking extendido, esos tokens cuentan como output (los más caros) y no aparecen en la respuesta visible. Útil para tareas complejas, derroche puro en tareas simples.

Detección: el response tiene usage.output_tokens mucho mayor al texto visible. Revisar config.
impacto típico
07 · fuga

Consultas repetitivas sin caché

Luciana recibe 40 veces por semana "¿cuál es mi saldo?" formulado de 15 maneras distintas. Cada vez se paga el LLM completo aunque la respuesta sea idéntica. Es la fuga más fácil de capturar en agentes de soporte B2B, donde el long-tail de consultas se concentra en pocos patrones.

Detección: embebé las últimas 1000 queries y agrupalas por similitud coseno > 0.92. Si el top-20 clusters concentra el 50%+ del tráfico, hay caché semántico esperando.
impacto típico
08 · fuga

Prompt drift silencioso

Alguien retoca el system prompt para arreglar un caso edge y sin querer agrega 600 tokens que se pagan en cada llamada futura. Con tres personas tocando el mismo agente, el prompt crece 15% por mes sin que nadie lo note hasta que llega la factura.

Detección: versionar el system prompt en git con tag por release. Graficar tokens del system prompt en el tiempo. Cualquier salto sin justificación de feature merece auditoría.
impacto típico
› nota sobre herramientas externas

¿Y los compresores tipo Headroom?

Hay proyectos como Headroom que se ofrecen como capa de compresión entre tu agente y el LLM: interceptan el contexto, comprimen lo redundante (sobre todo JSON, logs y código) y guardan el original local por si el modelo necesita el detalle. Funcionan bien en escenarios de alto volumen con tool outputs muy verbosos (logs SRE, builds, queries masivas). En flujos B2B moderados rinden poco, porque ahí los wins están en el diseño del agente, no en comprimir genéricamente. Se evalúa en la pestaña de decisión cuando corresponde.

› primer paso, siempre

Instrumentá antes de comprar

El API de Anthropic devuelve usage detallado en cada respuesta: input, output, cache_read, cache_write. Con loguear esos cuatro números a una tabla en 50 consultas reales, en una tarde sabés dónde está la fuga. Sin esa medición, optimizar es a ciegas.
