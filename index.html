Anatomía de una llamada al modelo

Cuando tu agente le manda una consulta al LLM, el modelo no recibe solo lo que escribió el usuario. Recibe un paquete que vos armaste: instrucciones, historial, resultados de herramientas, fragmentos recuperados. Todo eso se cobra como input, en cada turno.

Las ocho fugas

De dónde sale el gasto invisible

Cuando Alberto dice "esta consulta no era grande pero gastó un montón", casi seguro la fuga está en uno de estos ocho lugares. Cada uno se diagnostica distinto y se arregla distinto.

01 · fuga

System prompt inflado

Instrucciones largas, definiciones de tools con schemas verbosos, ejemplos few-shot, reglas de negocio. Se paga en cada llamada, aunque el usuario haya escrito tres palabras.

Detección: medí el campo usage.input_tokens en una llamada con input mínimo del usuario. Si ya da 5k+, es esto.

Fix: prompt caching — pero cuidado con dos trampas silenciosas:

Trampa 1 — punto de caché en bloque variable. Si ponés cache_control en el mensaje del usuario, una timestamp o cualquier bloque que cambia en cada request, el hash del prefijo nunca coincide: pagás write en cada llamada y no recibís ningún hit. La solución: poné cache_control en el último bloque que permanece idéntico entre requests — típicamente el cierre del system prompt, no el mensaje entrante.

Trampa 2 — mínimo de tokens por modelo. Sonnet 4.6 necesita al menos 1.024 tokens en el bloque a cachear. Si tu system prompt tiene menos, el caching se omite silenciosamente sin error. Verificá en la respuesta: si cache_creation_input_tokens y cache_read_input_tokens son ambos 0, no se está cacheando.
impacto típico
02 · fuga

Tool outputs verbosos

Una query SQL que vuelve con 500 filas, una API que retorna JSON con 80 campos por registro. El modelo recibe todo aunque solo necesite tres datos. Es lo más fácil de detectar y de arreglar.

Detección: logueá el tamaño en bytes de cada tool response antes de mandarla al modelo. Cualquier cosa arriba de 5KB merece revisión.
impacto típico
03 · fuga

Historial acumulado

Si la conversación tiene memoria y van 15 turnos, el turno 16 reenvía los 15 anteriores enteros. El costo total de la sesión crece cuadráticamente aunque cada turno parezca chico.

Detección: graficá input_tokens contra número de turno en una sesión real. Si sube en pendiente, es esto.
impacto típico
04 · fuga

RAG sobredimensionado

Top-20 chunks de 500 tokens = 10.000 tokens de contexto por consulta. Muchas implementaciones traen demasiado "por las dudas" cuando top-5 con buen umbral de similitud rinde igual o mejor.

Detección: mirá tu config de retriever. k arriba de 10 sin tuning explícito es sospechoso.
impacto típico
05 · fuga

Reintentos y loops silenciosos

Si el agente falla un tool call, el framework reintenta. Pagás dos, tres, cinco veces la misma consulta. LangGraph y Agno son los peores ofensores porque sus loops pueden iterar 8-10 veces antes de rendirse.

Detección: contá llamadas a /v1/messages por interacción del usuario — debería ser ~1, no 3-5.
impacto típico
06 · fuga

Thinking / extended reasoning

Si activaste modelos con thinking extendido, esos tokens cuentan como output (los más caros) y no aparecen en la respuesta visible. Útil para tareas complejas, derroche puro en tareas simples.

Detección: el response tiene usage.output_tokens mucho mayor al texto visible. Revisar config.
impacto típico
07 · fuga

Consultas repetitivas sin caché

Luciana recibe 40 veces por semana "¿cuál es mi saldo?" formulado de 15 maneras distintas. Cada vez se paga el LLM completo aunque la respuesta sea idéntica. Es la fuga más fácil de capturar en agentes B2B.

Detección: embebé las últimas 1000 queries y agrupalas por similitud coseno > 0.92. Si el top-20 concentra el 50%+ del tráfico, hay caché semántico esperando.
impacto típico
08 · fuga

Prompt drift silencioso

Alguien retoca el system prompt para arreglar un caso edge y sin querer agrega 600 tokens que se pagan en cada llamada futura. Con tres personas tocando el mismo agente, el prompt crece 15% por mes sin que nadie lo note.

Detección: versionar el system prompt en git con tag por release. Graficar tokens del system prompt en el tiempo. Cualquier salto sin justificación merece auditoría.
impacto típico
09 · fuga · nueva

Consultas sin clasificar por complejidad

"¿Cuántos registros hay?" consume ~9.500 tokens con Sonnet igual que "¿Cuál es la tendencia de los últimos 6 meses vs el año anterior?". El 60-70% de las consultas son simples y no necesitan el modelo caro.

Pasar todo por Sonnet cuando Haiku alcanza para la mayoría es como mandar un avión a buscar pan. La diferencia de costo es 12x.

Detección: clasificá manualmente tus últimas 50 consultas reales en SIMPLE (conteos, filtros directos, lookups) vs COMPLEJA (análisis, tendencias, correlaciones). Si más del 60% son simples, hay un pre-clasificador esperando.

Números concretos: 500 consultas/mes sin clasificar con Sonnet = ~USD 18. Con pre-clasificador Haiku (70% simples) = ~USD 3.20. Ahorro: 82%.
impacto típico
› nota sobre herramientas externas

¿Y los compresores tipo Headroom?

Hay proyectos como Headroom que interceptan el contexto, comprimen lo redundante y guardan el original local por si el modelo necesita el detalle. Funcionan bien en escenarios de alto volumen con tool outputs muy verbosos (logs SRE, builds, queries masivas). En flujos B2B moderados rinden poco, porque ahí los wins están en el diseño del agente, no en comprimir genéricamente. Se evalúa en el árbol de decisión cuando corresponde.

Instrumentá antes de comprar

El historial crece linealmente, pero el costo de la sesión crece cuadráticamente: en cada turno reenviás todos los anteriores. El turno 16 incluye los 15 anteriores completos. Una sesión de 20 turnos puede costar 5x más que una de 5 turnos con el mismo contenido por turno.

Cómo: a partir del turno 6 (ajustable), tomás los primeros 4 turnos y los mandás al modelo con un prompt "resumí esto en máximo 200 tokens preservando: decisiones tomadas, datos del cliente, contexto activo". Reemplazás esos 4 turnos por el resumen en el historial y mantenés los últimos 2 sin tocar.

Es lo que hace Claude Code internamente. Para un agente transaccional como el de la cerealera, la pérdida de fidelidad fina es totalmente aceptable.

Patrones arquitectónicos de optimización

Las fugas se tapan. Los patrones se diseñan. Estos tres no son correcciones de algo que salió mal — son decisiones de arquitectura que cambian el orden de magnitud del costo desde el principio.

Patrón 01

Pre-clasificador Haiku

Antes de cada consulta, un modelo liviano lee la pregunta y decide si es simple o compleja. Las simples se resuelven directamente a 12x menos costo. Las complejas pasan al flujo completo. En distribuciones reales de agentes B2B, el 60-70% de las consultas son simples.

Patrón 02

Export a archivo — el caso de las 120k filas

Cuando el cliente pide un volumen masivo legítimo (todos los movimientos del año, cierre mensual completo), la solución no es optimizar tokens — es sacar los datos del contexto del modelo completamente. El modelo orquesta, no lee. Las filas van directo a un archivo.

Flujo correcto

Lo que no debe pasar nunca

usuario → agente → LLM recibe 120k filas → responde
costo estimado: $0.18–$2+ por consulta, contexto desbordado

Lo que debe pasar

01 Usuario pide "todos los movimientos del año" LLM
02 LLM interpreta intención y arma el SQL LLM
03 Backend ejecuta COPY TO con streaming a archivo no LLM
04 Se genera CSV/Excel en Cloud Storage no LLM
05 LLM confirma: "listo, 120k filas en este link" LLM
Implementación

PostgreSQL + Cloud Run

En tu arquitectura GCP/Cloud Run, el endpoint de export strea el resultado de Postgres directamente a un archivo en Cloud Storage y devuelve un link de descarga firmado. El modelo nunca toca las filas.

COPY (SELECT * FROM movimientos
  WHERE fecha >= '2025-01-01')
TO STDOUT WITH (FORMAT CSV, HEADER);

COPY TO STDOUT es el método nativo de Postgres para exports masivos — significativamente más rápido que traer filas vía driver.

Patrón 04

Cache TTL extendido + Warmup

Dos técnicas de la API oficial que la mayoría de implementaciones ignora. Juntas eliminan el problema del TTL corto y la penalidad de latencia en la primera request del día.

TTL de 1 hora

Para conversaciones con pausas largas

El TTL por defecto es 5 minutos. Para la cerealera o el lab clínico, donde el usuario puede tardar 10-30 minutos en responder, el caché expira entre turnos y la siguiente request paga write de nuevo. La solución es TTL de 1 hora — sigue siendo entre 10x y 15x más barato que no cachear.

// en el system prompt
{
  "type": "text",
  "text": "...",
  "cache_control": {
    "type": "ephemeral",
    "ttl": "1h"
  }
}
Costo write TTL 1h = 2x precio base (vs 1.25x del TTL 5min).
Costo read = igual: 0.1x. El read es donde está el ahorro masivo.

Regla práctica: si las conversaciones tienen pausas >5min, usar 1h. Si el tráfico es continuo y denso, 5min alcanza.
Cache Warmup

Eliminar la penalidad del primer turno

El primer request de cada día paga cache write (más caro) y tiene latencia mayor. Con max_tokens: 0 podés "precalentar" la caché al arrancar la app — sin generar output, sin cobrar tokens de salida. El primer usuario real ya encuentra la caché caliente.

// al arrancar la app o cada N minutos
$client->messages->create(
  model: 'claude-sonnet-4-6',
  maxTokens: 0, // ← clave
  system: [[ /* tu system prompt */
    'cache_control' => ['type' => 'ephemeral']
  ]],
  messages: [['role' => 'user',
    'content' => 'warmup']]
);
// response: content=[], stop_reason="max_tokens"
// 0 output tokens facturados
Importante: el cache_control va en el system prompt, NO en el mensaje "warmup". Si va en el mensaje, la caché se asocia a ese placeholder y el request real no la encuentra.

Para TTL 5min: repetir el warmup cada 4-5 minutos si el gap entre usuarios puede ser largo. Para TTL 1h, una vez al arrancar alcanza.
